# WS Project Context Manager (ws-pcm)

Универсальная система управления контекстом проекта. Работает для ЛЮБОГО типа: разработка, маркетинг, брендинг, аналитика, стратегия сайта.

## Принципы

- **80/20** — минимум документов, максимум пользы
- **Single Source of Truth** — docs/MainDoc.md = главный документ
- **Context > Motivation** — быстрое вхождение за ≤5 минут через T-last
- **Три аудитории** — Team (Markdown), AI agents (JSON), Clients (branded .docx/.pdf)
- **Data Verification Layer (DVL)** — каждое утверждение в документах должно иметь метку источника. Никакие данные не принимаются «на веру» — всё атрибутировано
- **Activity Log** — хронологический журнал действий по проекту. Каждая сессия фиксируется

## Правило: PCM = Single Source of Truth (Memory Sync)

**Принцип:** Проектное состояние ВСЕГДА должно быть актуальным в PCM-файлах в рабочей директории проекта. `MEMORY.md` используется только для сессионных заметок AI (уроки, стратегии работы с файлами, паттерны ошибок).

**Что куда записывать:**

| Тип информации | Куда | Пример |
|----------------|------|--------|
| Текущий этап проекта, приоритеты, дедлайны | `.ai/context.json` → `currentFocus` | stage, status, deadline, blockers |
| Где остановились, что дальше | `.ai/t-last.json` | whereStopped, nextStep, deferred |
| Хронология действий | `docs/ActivityLog.md` | Новая запись с датой, действиями, результатами |
| Результаты встреч, решения клиента | `context.json` → секция `meetingResults_*` + `ActivityLog.md` | Подтверждённые факты, открытые вопросы |
| Обновления данных клиента | `context.json` → `clientData` + обновить релевантные Spec-файлы | Trust Claims, Partner, статусы |
| Уроки работы с проектом (AI-specific) | `MEMORY.md` (локально) | Стратегия чтения больших файлов, паттерны ошибок |

**Обязательное правило при закрытии сессии:**
Перед закрытием сессии (`/compact`, "закрывай сессию") — убедиться, что ВСЕ изменения состояния проекта отражены в PCM-файлах в рабочей директории, а не только в `MEMORY.md`.

**Причина:** Проект может быть открыт другим участником команды или другой AI-сессией. Если актуальные данные только в `MEMORY.md` — они потеряны для всех кроме текущей сессии.

## Правило: Workspace Prefix — Атрибуция записей

**Принцип:** Каждая запись в PCM-файлах проекта (ActivityLog, t-last, комментарии) должна содержать **префикс рабочего пространства** — две заглавные буквы инициалов владельца workspace. Это позволяет в shared-проектах видеть, кто (из какого воркспейса) сделал запись.

**Формат:** Префикс = первые буквы имени и фамилии владельца воркспейса, uppercase.
- Mihail Taube → `MT`
- Alexander Wirt → `AW`

**Где применять:**

**ActivityLog.md** — в заголовке записи:
```markdown
### 2026-03-05 11:30 [MT] — Meeting 03.03 analysiert, PCM aktualisiert
```

**t-last.json** — поле `author`:
```json
{
  "timestamp": "2026-03-05T11:30:00+01:00",
  "author": "MT",
  "session": "..."
}
```

**context.json** — при обновлении `currentFocus`, добавить `lastUpdatedBy`:
```json
"currentFocus": {
  "lastUpdatedBy": "MT",
  "lastUpdatedAt": "2026-03-05T11:30:00+01:00"
}
```

**Условие применения:**
- Если в настройках воркспейса (preferences) задано имя пользователя — вычислить префикс автоматически и использовать
- Если имя не задано — записи создаются **без** префикса (правило не ломается, просто нет атрибуции)
- При первом обнаружении имени — предложить зафиксировать префикс

**Зачем:** В проекте, где работают несколько воркспейсов, по ActivityLog сразу видно, кто какую запись сделал. Это критично для ответственности и трейсабилити решений.

## Команды

| Аргумент | Действие |
|----------|----------|
| `init` | Auto-Bootstrap: создать docs/ + .ai/ + client/ структуру |
| `resync` | **Полная актуализация существующего проекта:** анализ всех файлов и источников, заполнение PCM-структуры, архивация устаревших файлов |
| `update "описание"` | Обновить ContextCard + T-last + Activity Log после сессии |
| `snapshot` | Создать еженедельный снапшот в ContextCard |
| `status` | Показать текущий статус проекта |
| `client "тип"` | Сгенерировать брендированный документ для клиента |
| `dvl` | Показать/обновить Data Verification Layer (реестр данных с источниками) |
| `log "действие"` | Добавить запись в Activity Log |

Примеры вызова: `/pcm init`, `/pcm resync`, `/pcm update "добавил авторизацию"`, `/pcm dvl`, `/pcm log "прочитал Kundenbrief"`

## Structure Health Check (выполняется ПЕРЕД любой командой)

> **Когда:** При КАЖДОМ вызове PCM (`init`, `resync`, `update`, `status`, `snapshot`, `dvl`, `log`, `client`).
> **Цель:** Убедиться, что структура проекта соответствует стандарту PCM. Если нет — предложить исправление ДО выполнения команды.
> **Время:** < 10 секунд (только проверка наличия файлов, не чтение содержимого).

### Чеклист проверки

| # | Проверка | Ожидание | Если нарушено |
|---|----------|----------|---------------|
| 1 | `ContextCard.md` в **корне** проекта | Файл существует в `PROJECT_ROOT/ContextCard.md` | ⚠️ Если в `docs/` или другом месте → предложить переместить в корень |
| 2 | `docs/MainDoc.md` | Файл существует | ⚠️ Предложить создать через `resync` |
| 3 | `docs/DataRegistry.md` | Файл существует | ⚠️ Предложить создать |
| 4 | `docs/ActivityLog.md` | Файл существует | ⚠️ Предложить создать |
| 5 | `.ai/AGENTS.md` | Файл существует | ⚠️ Предложить создать |
| 6 | `.ai/context.json` | Файл существует, валидный JSON | ⚠️ Предложить создать |
| 7 | `.ai/t-last.json` | Файл существует, валидный JSON | ⚠️ Предложить создать |
| 8 | DVL-категории актуальны | `AGENTS.md` содержит `✅ CLIENT` в приоритетах и правило VERBATIM | ⚠️ Предложить обновить AGENTS.md |
| 9 | `integrations` в `context.json` заполнены | Хотя бы 1 интеграция имеет непустые значения | ⚠️ Запустить автозаполнение (см. Integrations Auto-Discovery) |

### Логика

1. **Все проверки пройдены** → выполнить команду как обычно.
2. **Есть нарушения** → показать список нарушений пользователю:
   ```
   ⚠️ PCM Structure Check — обнаружены проблемы:
   - ContextCard.md находится в docs/ вместо корня проекта
   - .ai/AGENTS.md не содержит правило VERBATIM

   Исправить перед продолжением? (да/нет)
   ```
3. **Пользователь говорит «да»** → исправить, затем выполнить команду.
4. **Пользователь говорит «нет»** → зафиксировать в ActivityLog как риск, выполнить команду.

### Исключения
- Команда `init` — health check пропускается (структура ещё не создана).
- Если PCM-структура полностью отсутствует (нет ни одного из 7 файлов) → предложить `init` или `resync` вместо текущей команды.

## Integrations Auto-Discovery

> **Когда:** Health Check обнаружил пустую секцию `integrations` в `context.json`, или при `resync`/`update` если интеграции ещё не заполнены.
> **Цель:** Автоматически найти и заполнить связи проекта с внешними системами, чтобы AI-агент при входе в проект сразу знал, к каким ресурсам обращаться.

### Источники обнаружения

Искать данные об интеграциях **в порядке приоритета:**

| # | Где искать | Что извлечь | Пример |
|---|-----------|-------------|--------|
| 1 | **Рабочая директория проекта** | Google Drive folder ID из URL или метаданных | `drive.folderId` |
| 2 | **`docs/MainDoc.md`** → секция Links | Ссылки на GitHub, Figma, Miro, Vercel, Bitrix24 | Все интеграции |
| 3 | **`docs/DataRegistry.md`** → Реестр источников | CRM-ссылки, Google Docs, Miro-доски | `crm`, `board`, `design` |
| 4 | **`ContextCard.md`** | Упоминания внешних систем | Все интеграции |
| 5 | **Файлы проекта** | `package.json` → repo URL, `.env` → API endpoints, `vercel.json`, `CLAUDE.md` | `repo`, `hosting` |
| 6 | **MEMORY.md** (сессионная память) | Ссылки на системы, использованные в предыдущих сессиях | Все интеграции |
| 7 | **Имя папки проекта + активные источники** | Поиск по CRM (Bitrix24), если есть совпадение по имени проекта | `crm` |

### Алгоритм

1. **Прочитай `context.json` → `integrations`.** Если секция отсутствует — создай пустую по шаблону.
2. **Для каждой пустой интеграции** — пройди по источникам обнаружения (таблица выше).
3. **Нашёл данные** → заполни поля, запиши в `context.json`.
4. **Не нашёл** → оставь пустым (не выдумывай).
5. **Покажи результат пользователю:**
   ```
   🔗 Integrations Auto-Discovery:
   ✅ drive: Google Drive → Projekte/RenoTec (folder: 10j8t...)
   ✅ crm: Bitrix24 → Deal #1234
   ✅ repo: GitHub → 2penguins/renotec-website
   ⬜ design: не найдено
   ⬜ board: не найдено
   ✅ hosting: Vercel → renotec.vercel.app

   Подтвердить? (да / исправить)
   ```
6. **Пользователь подтвердил** → записать в `context.json`.
7. **Пользователь исправил** → обновить с учётом правок.

### Когда запускается автоматически
- При `resync` — шаг 4 (финализация), после создания `context.json`
- При `init` — шаг 5 (заполнение файлов), при создании `context.json`
- При Health Check — если обнаружено нарушение #9

### Когда НЕ запускается
- При `update`, `log`, `snapshot` — если интеграции уже заполнены (хотя бы 1 непустая)
- Если пользователь ранее явно отказался от заполнения

## init — Auto-Bootstrap

При вызове `/pcm init`:

**1. Проверь рабочую директорию на наличие существующих файлов.**

Если в папке уже есть файлы проекта (документы, контент, анализы, экспорты и т.д.) — это НЕ новый проект. В этом случае:
- **Сообщи пользователю**, что обнаружены существующие файлы (покажи краткий обзор: сколько файлов, какие типы, есть ли уже PCM-структура)
- **Предложи `resync`** вместо `init`: *«Обнаружены существующие файлы проекта. Рекомендую `/pcm resync` — он проанализирует все данные и заполнит структуру PCM без потери существующей информации.»*
- Если пользователь всё равно хочет `init` — предупреди, что новая структура будет создана поверх существующей, и предложи сначала архивировать старые файлы

Если папка пуста или содержит только базовые файлы:

**2. Спроси у пользователя (ВСЕ пункты обязательны):**
- Название проекта
- Тип (`website-strategy` / `development` / `marketing` / `branding` / `analytics` / `other`)
- Цель проекта (1 предложение)
- Владелец (Owner)

**3. ⚡ ОБЯЗАТЕЛЬНО: Сбор источников данных**

> Этот шаг НЕЛЬЗЯ пропускать. Без полных исходных данных структура проекта будет неполной, а это приведёт к потере контекста и ошибкам в работе.

**Спроси у пользователя:**

*«Прежде чем создавать структуру проекта, мне нужно собрать ВСЕ доступные источники данных. Это критически важно — чем больше данных я получу сейчас, тем полнее будет контекст проекта.*

*Есть ли что-то из следующего?*
- *📁 Файлы или папки с материалами проекта (Google Drive, локальные файлы)*
- *📋 CRM-данные (Bitrix24, HubSpot — сделка, контакт, компания)*
- *📄 Документы от клиента (бриф, ТЗ, концепт, заполненные опросники)*
- *💬 Переписка или чаты по проекту (Telegram, WhatsApp, email)*
- *🔗 Ссылки на предыдущую работу (ChatGPT, другие AI-агенты, Notion, Miro)*
- *📞 Записи встреч или созвонов (Plaud, Otter, заметки)*
- *🌐 Существующий сайт или прототип*
- *📊 Аналитика, исследования, конкурентный анализ*
- *Что-то ещё, что может содержать данные о проекте?»*

**Дождись ответа пользователя.** Если пользователь говорит «нет» или «потом» — зафиксируй это в ActivityLog и продолжай, но пометь в ContextCard как риск: *«Источники данных не собраны при init — возможна неполнота контекста»*.

**⚡ ОБЯЗАТЕЛЬНО: Зарегистрируй ВСЕ источники** в `docs/DataRegistry.md` → секция «Реестр источников» (см. формат в описании `resync`). Каждая ссылка, каждый файл, каждый чат — всё должно быть записано с типом и статусом прочтения.

Если пользователь предоставил ссылки/файлы — **прочитай их ВСЕ** перед заполнением структуры (шаг 5). По мере чтения обновляй статус источника: `⏳ Ожидает` → `✅ Прочитан`.

**4. Создай CORE структуру:**

```
PROJECT_ROOT/
├── ContextCard.md                # ⭐ ТОЧКА ВХОДА. Читай это ПЕРВЫМ.
│                                 # Текущий статус, T-last, Weekly Snapshot
│
├── docs/                         # Team Layer (Markdown)
│   ├── MainDoc.md               # Source of Truth — все данные проекта
│   ├── PSP.md                   # Project System Prompt
│   ├── DataRegistry.md          # DVL: реестр данных с источниками
│   └── ActivityLog.md           # Журнал действий по проекту
│
├── .ai/                          # AI Layer (Structured)
│   ├── AGENTS.md                # Инструкции для AI агентов
│   ├── context.json             # Метаданные проекта (machine-readable)
│   └── t-last.json              # Текущий T-last (machine-readable)
│
├── client/                       # Client Layer (branded docs only)
│
└── materials/                    # Исходные материалы, брифы
```

> **Почему ContextCard.md в корне:**
> Когда кто-то (человек или AI) открывает папку проекта, первое что он видит — `ContextCard.md`.
> Это единственная точка входа: текущий статус, что делать дальше, где что лежит.
> Все остальные документы — в подпапках, чтобы не захламлять корень.

**5. Заполни файлы по шаблонам ниже**, подставив данные от пользователя и из собранных источников (шаг 3).

**6. Data Verification Layer — ОБЯЗАТЕЛЬНО при init:**
- Прочитай ВСЕ источники, собранные в шаге 3
- Заполни `docs/DataRegistry.md` — каждый факт получает метку источника
- Любое утверждение без подтверждённого источника → `⚠️ AI`, в секцию «Неподтверждённые данные»
- **Проверяй противоречия между источниками** и фиксируй расхождения

**7. Activity Log — первая запись:**
- Создай `docs/ActivityLog.md` с первой записью о bootstrap проекта
- Укажи: какие файлы/источники прочитаны, какие данные собраны, какие решения приняты

## resync — Полная актуализация существующего проекта

При вызове `/pcm resync`:

> **Когда использовать:** Проект уже существует, есть файлы и данные (возможно разрозненные, в разных форматах, от разных инструментов), но PCM-структура отсутствует, неполная или устарела. Resync собирает ВСЕ данные в единую структуру без потерь.

### Шаг 0: ⚡ Сбор источников (ОБЯЗАТЕЛЬНО)

> Этот шаг выполняется ПЕРВЫМ, до любого анализа файлов. Цель — узнать у пользователя обо ВСЕХ местах, где могут быть данные проекта.

**Спроси у пользователя:**

*«Перед тем как я начну анализ, мне нужно знать обо ВСЕХ источниках данных по этому проекту. Работа могла вестись в разных местах и инструментах — важно ничего не упустить.*

*Подскажи, пожалуйста:*
1. *Есть ли файлы или папки помимо текущей рабочей директории? (Google Drive, другие папки, облачные хранилища)*
2. *Есть ли CRM-данные? (Bitrix24, HubSpot — сделка, контакты, компания)*
3. *Велась ли работа в других AI-инструментах? (ChatGPT, другие агенты, Notion AI — ссылки на чаты/проекты)*
4. *Есть ли документы от клиента? (бриф, ТЗ, концепт, заполненные опросники, записи встреч)*
5. *Есть ли переписка по проекту, которую стоит учесть? (чаты, email)*
6. *Есть ли ссылки на предыдущие сессии или транскрипты работы?*
7. *Что-то ещё, что может содержать важные данные?»*

**Дождись ответа.** Все предоставленные ссылки и источники будут использованы в шагах 1-2.

**⚡ ОБЯЗАТЕЛЬНО: Зарегистрируй ВСЕ источники.**
Сразу после получения ответа — создай или обнови секцию **«Реестр источников»** в `docs/DataRegistry.md`:

```markdown
## Реестр источников

> Все места, откуда были получены данные проекта. Каждый источник = потенциальный первоисточник для DVL-атрибуции.

| # | Источник | Тип | Ссылка / путь | Дата добавления | Статус |
|---|----------|-----|---------------|-----------------|--------|
| 1 | Рабочая папка проекта | 📁 Файлы | {путь} | {дата} | ✅ Прочитан |
| 2 | {название} | {тип} | {ссылка} | {дата} | ⏳ Ожидает / ✅ Прочитан / ❌ Нет доступа |
```

Типы источников: `📁 Файлы` / `📋 CRM` / `📄 Google Doc` / `💬 Чат/переписка` / `🤖 AI-сессия` / `📞 Запись встречи` / `🌐 Сайт` / `📊 Аналитика`

По мере чтения источников — обновляй статус в реестре. Это позволит в будущем **проследить, откуда именно** была взята каждая единица данных.

Если пользователь говорит «всё в папке» — ок, переходи к шагу 1. Но если позже обнаружишь ссылки на внешние источники внутри файлов (Google Docs, Bitrix24, ChatGPT) — **добавь их в реестр источников, сообщи пользователю и спроси, нужно ли их тоже прочитать**.

### Шаг 1: Инвентаризация

1. **Рекурсивно просканируй** всю рабочую директорию проекта
2. Для каждого файла определи:
   - Тип: `клиентский документ` / `стратегия` / `контент` / `анализ` / `ТЗ` / `экспорт` / `медиа` / `код` / `другое`
   - Дата создания/изменения
   - Ключевые данные внутри (кратко)
   - Актуальность: `актуален` / `устарел` / `дубликат` / `не относится к проекту`
3. Если есть внешние источники (Bitrix24, Google Drive, Google Docs) — **прочитай их тоже**
4. Покажи пользователю **сводку инвентаризации:**
   - Всего файлов / по типам
   - Обнаруженные дубликаты
   - Файлы, не относящиеся к проекту
   - Существующая PCM-структура (если есть)

### Шаг 2: Анализ и извлечение данных (ИНКРЕМЕНТАЛЬНО)

**КРИТИЧЕСКИ ВАЖНО:** Не сокращать, не упускать деталей. Каждый факт от клиента должен быть зафиксирован.

> **⚠️ ЗАЩИТА ОТ ПОТЕРИ КОНТЕКСТА:**
> При большом количестве файлов и источников контекстное окно может переполниться, и ранее прочитанные данные будут сжаты или потеряны.
>
> **Поэтому данные записываются В ФАЙЛЫ СРАЗУ, а не «потом».** Алгоритм:
> 1. Прочитал файл → извлёк данные → **сразу записал** в DataRegistry.md / MainDoc.md / ActivityLog.md
> 2. Перешёл к следующему файлу → прочитал → **дополнил** существующие записи
> 3. Если новый файл **противоречит** ранее записанному — **обнови** запись, указав оба источника и дату каждого
> 4. Если новый файл **дополняет** — **добавь** к существующей записи
> 5. Более поздний по дате документ → вероятно актуальнее. Но первичный источник (Kundenbrief) → приоритетнее AI-производного
>
> **Никогда не откладывай запись «на потом».** Каждый прочитанный файл = немедленное обновление проектных документов.

**Порядок чтения файлов (по приоритету):**
1. Первичные клиентские документы (Kundenbrief, брифы, записи встреч) — **читай и записывай первыми**
2. Утверждённые спецификации (ТЗ, SPEC, мастер-документы) — дополняй
3. Аналитика и исследования (конкурентные анализы, аудиты) — дополняй
4. Контент и производные документы (тексты, черновики) — дополняй
5. AI-сгенерированные файлы — **проверяй** каждый факт, не доверяй автоматически

**Из каждого файла извлеки:**
- Факты о компании/проекте
- Принятые решения (с датой!)
- Числовые показатели и утверждения
- Контактные данные
- Ссылки и референсы

**Каждому факту присвой метку источника** (DVL): `📋 REG` / `📝 BRIEF` / `📄 SPEC` / `💬 CHAT` / `🔍 RESEARCH` / `⚠️ AI`

## Data Verification Layer (DVL)

### Принципы DVL

1. **Каждое утверждение = источник.** Если данные не имеют метки источника — они не подтверждены.
2. **Приоритет источников:** `📋 REG` > `📄 SPEC` > `📝 BRIEF` > `✅ CLIENT` > `💬 CHAT` > `🔍 RESEARCH` > `⚠️ AI`
3. **AI-данные запрещены к публикации** без явного перевода в `✅ CLIENT` клиентом.
4. **Противоречия разрешаются по приоритету:** если Handelsregister говорит одно, а Kundenbrief другое — верен реестр. Если SPEC и BRIEF противоречат — уточнять у клиента.
5. **Первичные документы клиента** (Kundenbrief, заполненные опросники, подписанные ТЗ) — ключевой источник бизнес-данных, которые невозможно получить из открытых реестров.
6. **Миграция меток:** При изменении DVL-категорий (напр. `💬 CHAT` → `✅ CLIENT`) — проверь **ВСЕ** файлы проекта на оставшиеся упоминания старой метки, включая ссылки `**Quelle:**` в MainDoc и других документах.

### При `init`:
- Прочитай ВСЕ доступные первичные источники (Kundenbrief, Handelsregister, SPEC, материалы в папке проекта)
- Заполни DataRegistry.md с атрибуцией КАЖДОГО факта
- Все утверждения без источника → пометь `⚠️ AI` и добавь в секцию "Неподтверждённые данные"

### При `update`:
- Проверь, появились ли новые данные в сессии
- Обнови DataRegistry.md соответствующими записями
- Если данные были подтверждены клиентом → смени метку на `✅ CLIENT`

### При `dvl`:
1. Прочитай `docs/DataRegistry.md`
2. Покажи сводку:
   - Всего утверждений
   - По категориям источников (сколько `📋 REG`, сколько `⚠️ AI` и т.д.)
   - Список неподтверждённых данных, требующих действий
3. Если указан аргумент `dvl audit` — пройдись по ВСЕМ файлам проекта и проверь, что каждое утверждение есть в реестре

## update — Обновление контекста после сессии

При вызове `/pcm update "описание"`:

1. Прочитай текущие `ContextCard.md` и `.ai/t-last.json`
2. Проанализируй текущую сессию (все изменения, решения, результаты)
3. Сгенерируй новый T-last (6 точек):
   - **Где остановились** — последний файл/секция над которыми работали
   - **Следующий микро-шаг** — конкретное действие ≤15 мин
   - **Отложено и почему** — что решили не делать сейчас
   - **Почему важно** — 1 строка мотивации
   - **Риск задержки** — что произойдёт если откладывать
   - **If-Then** — условие → действие для следующего шага
4. Обнови `ContextCard.md` → секцию T-last
5. Обнови `.ai/t-last.json` (структурированные данные)
6. Если изменились метаданные проекта → обнови `.ai/context.json`
7. **Добавь запись в `docs/ActivityLog.md`** — дата, описание сессии, затронутые файлы, использованные источники, принятые решения
8. **Обнови `docs/DataRegistry.md`** — если в сессии появились новые данные или были подтверждены существующие
9. Покажи diff изменений пользователю
